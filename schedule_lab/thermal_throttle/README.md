# 实时热降频监测(Thermal Throttling)

实时采 GPU **温度 + SM 频率**(+功率/利用率/NVML 降频原因),在某一时刻**突然加满负载**把芯片烧热,观察温度冲过安全阈值后**系统自动降频**把温度压回来的全过程。

```
thermal_throttle/
├── monitor_throttle.py   # 20Hz 采样 + 1Hz 实时打印;基线→满载GEMM→冷却
├── plot_throttle.py      # 画温度+频率双轴时间序列
├── throttle.csv          # 原始遥测(每 50ms 一行)
├── meta.json             # 阈值、加载窗口、峰值温度、最低频率
└── fig_thermal_throttle.png
```

运行:`CUDA_VISIBLE_DEVICES=0 python3 monitor_throttle.py && python3 plot_throttle.py`(无需 sudo)。

## 实验设计
- **不锁频**——目的就是观察系统自己的热管理(锁频反而看不到)。
- 负载:持续 `8192² fp16 GEMM`,把功率打到墙(~278W)、快速升温。
- 时间线:基线 15s(怠速)→ **满载 90s** → 冷却 50s。
- 用 `nvmlDeviceGetCurrentClocksThrottleReasons` 直接读"降频原因",证明是**热降频**而非别的。

## 观测结果(Tesla V100-DGXS-32GB)
阈值:**最大工作温度 83°C**、硬降频 87°C、关机 90°C。

| 阶段 | 温度 | SM 频率 | 功率 |
|---|---|---|---|
| 基线(怠速) | 54°C | 135 MHz | 42 W |
| 加载瞬间(t≈18s) | 64→**冲高** | boost **1305 MHz** | 278 W |
| **撞 83°C(t≈23s)** | 83°C | 开始掉:1305→1215 | 268 W |
| 稳态(满载持续) | **钉在 82–83°C** | 一路降到 **~250–480 MHz** | ~120 W |
| 卸载后冷却 | 83→73°C 回落 | 恢复怠速 | 68 W |

**关键现象**:温度一到 **83°C(最大工作温度)**,NVML 立刻报 `HW/SW Thermal Slowdown`(全程 885 个降频采样),时钟从 boost 的 **1305 MHz 被砍到最低 247 MHz(降 ~80%)**,功率 278→120W,从而把温度稳稳压在 83°C——**完全是硬件热管理闭环**:温度越界 → 自动降频 → 温度回落 → 维持在安全线。硬降频 87°C 全程没触及(管理机制在 83°C 就把它拦住了)。

> 这解释了之前 prefill/decode 持续重载时的"时钟塌缩":并非锁频失败,而是**热管理在 83°C 主动降频**,即使我们 `-lgc` 锁了频也压不住(硬件级热保护优先级最高)。这也是为什么这张卡持续吞吐远低于 boost 频率下的理论值。
