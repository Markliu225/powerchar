# 从"固定 P:D 比例"到"按应用分类的工作负载" —— 一份(规划阶段的)思考文档

## 这份文档在想什么

先说清楚我们站在哪个阶段:这是**规划(planning)**,不是运行时(runtime)。也就是说,我们要在部署之前、离线地,根据一份对负载的**预期/预测**,一次性决定整个机群该怎么搭——总共要多少机架、每种应用分到几个机架、每个机架里 prefill 和 decode 卡怎么劈分、每张卡限到多少功率。产出是一张**静态的容量蓝图**,而不是去响应每一秒钟的流量变化。怎么在线路由、怎么动态调度,是另一个阶段的事,这份文档不碰。

现在的 `solve.py` 把规划的输入压成一个数字:prefill 与 decode 的 token 长度比例 `P:D`。这在数学上很干净,但它等于假设"全世界的请求都长一个样"。真实要规划的流量不是这样:它来自不同的应用——有人聊天,有人用 RAG 问文档,有人让模型写长文,有人批量做摘要或分类。每一种应用的 prefill 长度和 decode 长度都不是一个固定值,而是一个范围;而且一个机房通常要同时承载这许多种应用。这份文档想把"规划阶段该怎么对待这种现实"想清楚,并且会用到一个关键假设:**我们可以用 router 让每一个机架只服务一种应用**。注意,router 本身是运行时的机制,但它对规划的意义是——我们在**设计时**就可以假定机架是按应用专用的,从而把每种应用当成一个独立的池子来分别规划。

## 真实负载长什么样:每种应用是一块区域,不是一个点

先把"按应用分类"具体化。下表是几类典型应用的经验长度区间(token),`L_p` 是 prefill(提示)长度,`L_d` 是 decode(生成)长度:

| 应用 | prefill `L_p` | decode `L_d` | 典型 P:D | 上下文 C | 延迟 SLO |
|---|---|---|---|---|---|
| 对话 Chat | 50–500 | 100–800 | ~1:2(decode 重) | 短 | 交互 |
| 长文生成(写作) | 50–500 | 1000–4000 | ~1:10 | 中 | 半交互 |
| RAG / 文档问答 | 1000–8000 | 50–400 | ~10:1(prefill 重) | 长 | 交互 |
| 摘要 Summarization | 2000–32000 | 100–500 | ~30:1 | 很长 | 批 |
| 代码补全 Code | 200–2000 | 20–200 | ~10:1 | 中 | 交互(TTFT 极敏感) |
| 翻译 Translation | 100–2000 | 100–2000 | ~1:1 | 中 | 批 / 交互 |
| 分类 / 信息抽取 | 100–4000 | 1–10 | ≈ prefill-only | 中 | 批 |

关键是看最后两列:不同应用不只是"比例不同",它们的**上下文长度**差着两个数量级,**延迟要求**也从"逐字必须跟手"到"跑完就行"天差地别。如果把这些请求画在一张 `(L_p, L_d)` 平面上,旧模型相当于只承认一条过原点的射线(斜率就是 D/P),而真实世界是好几块散落在不同位置的二维区域,一份要规划的负载就是这些区域按各自预期请求量叠起来的一团点云。"比例"只是这团点云的质心方向,它既丢掉了每块区域的形状,也丢掉了上下文和延迟这两个隐藏维度。

## 关键假设:router 让每个机架只服务一种应用

现在引入那个简化假设,并且要在规划的语境下理解它。我们在规划时假定:机房会用一个 router,把请求按应用类型分到不同的机架去——聊天请求只进"聊天机架",摘要请求只进"摘要机架",RAG 请求只进"RAG 机架"。换句话说,我们**不打算**设计那种"一个机架同时迁就好几种应用"的通用机架,而是打算把机架按应用**专用化**。

这一步为什么重要,要看它在规划时消掉了什么。如果允许机架混跑,那么规划时就得面对最难的情形:同一个机架、甚至同一个 decode 批次里,混着 256 上下文的聊天请求和 8000 上下文的 RAG 请求——这两者每个 token 的访存成本差几十倍,根本没法用同一条曲线、同一个效率甜点去描述,任何"平均"都是误导,规划也就无从下手。一旦设计上规定每个机架专供一种应用,这种异质性就消失了:每个机架内部所有请求的上下文长度落在同一个量级,延迟要求一致,长度也是同一个应用的分布。原本纠缠在一起的复杂度,被拆成了一池一池彼此独立、各自同质的机架。

于是整个规划问题自然裂成上下两层,而且这两层可以分开解。一层是**机架内**:给定一种应用,怎么劈分 prefill / decode 卡、每张卡限到多少功率,也就是这种应用的"机架配方"。另一层是**机架间**:每种应用按它的预期需求,该分到几个机架。能把这两层解耦,正是 router 假设在规划上最大的价值——它让我们可以为每一种应用单独设计一套机架,再把机房按应用切成几块,而不必在一个机架里同时满足互相冲突的需求。

## 机架内:单应用其实就是我们已经解过的那个问题

一个专供单一应用的机架,上下文 C 基本是个常数(聊天就都短、摘要就都长),延迟 SLO 是同一档,长度虽然有范围但属于同一个应用的连贯分布。在这种条件下,我们之前那套解法——按有效比例劈分 prefill / decode 卡、让每张卡跑在效率甜点——可以原封不动地拿来用,只需要把两个旋钮换成这个应用自己的取值:decode 的吞吐天花板用这个应用的上下文 C 来算,功率 cap 的下界用这个应用的延迟 SLO 来定(下一节细说这两件事)。

那"范围"在规划时怎么处理?它没有消失,但因为现在面对的是单一应用的一个连贯分布,它只通过两件事进来,而且都是设计时能定下来的。一是它的**均值**,给出这个机架自己的有效比例 `R = E[L_p] : E[L_d]`,决定 prefill 和 decode 卡怎么劈——这一步就是把派生出来的有效比例喂给现有的 `solve.py`。二是它的**离散程度**(方差、长尾),决定我们在规划时要给这个池子留多少容量余量:如果一种应用的长度分布很散,就得按偏高的分位数来配卡,而不是按均值,免得一规划完就发现常态化地不够用。这两件事都是静态的设计选择,不需要任何运行时的反应。换句话说,单一应用内部的"范围"是个相对温和的规划问题:按合适的分位数留一点余量就能覆盖,远没有跨应用混合那么棘手。router 的作用,正是把"棘手的跨应用混合"换成了"温和的单应用余量估计"。

## 不同应用的机架,配方完全不一样

虽然每个专用机架内部都好规划,但不同应用的机架,规划出来的配方会很不一样,差别主要来自两处。

第一处是**上下文长度**,这也是最被低估的一项。decode 是访存受限的:每生成一步要把全部模型权重读一遍,再加上批次里所有序列的 KV cache,而 KV 正比于上下文长度。结果是单卡 decode 吞吐的天花板正比于 1/C。用我们这张卡实测的"C=256 → 721 tok/s"反推,可以得到一个很直观的换算 `T_max(C) ≈ 721 × 256/C`:

| 上下文 C | 单卡 decode 天花板 |
|---|---|
| 256(聊天) | ~721 tok/s |
| 2 000(代码 / 短 RAG) | ~92 tok/s |
| 8 000(长 RAG) | ~23 tok/s |
| 32 000(摘要) | ~6 tok/s |

也就是说,一个聊天机架的 decode 天花板有七百多 tok/s,而一个摘要机架的 decode 天花板只剩个位数——同样产出一个 token,摘要机架要贵上百倍。这直接决定了规划出来的配方:摘要机架必须把绝大多数卡都堆在 decode 上,而且整条 decode 能效曲线被压得很低;聊天机架则相对均衡。所以"decode 机架"这个说法,在不同应用下根本不是同一种东西,各自的吞吐和经济性都得分开算。我们之前算出来的那个回本(约 6.5 个月),其实只是"聊天类机架"的一个特例。

第二处是**延迟要求**。功率 cap 压低时钟,会同时拉长首 token 延迟(prefill 决定)和逐字延迟(decode 每步决定)。聊天和代码补全是交互应用,有硬性的延迟红线,所以规划它们的机架时,功率 cap 存在一个下界,不能为了能效一路压到甜点,否则交付出去就违约;而离线摘要、批量分类这类没有紧延迟约束的应用,机架可以放心压到效率甜点,把每瓦 token 榨到最高。于是同样是"限功率",交互机架在规划上是"在不违约的前提下尽量省",批处理机架才是"一路省到底"。

把这两处合起来看:每一种应用机架都有它自己的一整套配方——自己的 prefill : decode 劈分、自己的功率 cap、由上下文决定的自己的 decode 天花板,因而自己的吞吐曲线和回本周期。router 假设的意义,就是让我们在规划时可以为每一种应用单独配一套这样的配方。

## 机架间:把机房静态地切给各种应用

机架配方定好之后,规划层面剩下的就是一个**切分**问题:给定每种应用的预期需求(用预测的请求量、或换算成预测的 prefill / decode token 量来表示),按需求把机房里的总机架数分给各种应用,每种分到几个专用机架。这是一个一次性的、静态的设计产出——一份"X 个聊天机架 + Y 个 RAG 机架 + Z 个摘要机架"的蓝图,针对的是规划时所用的那份需求预测。如果将来长期的负载结构确实变了,那是下一轮重新规划的事,而不是机架在运行中自己挪来挪去。

当然这套好处不是白来的,有它在规划上的代价。最主要的一条是**粒度**:专用化要划算,得让每种应用的量都足够大、能基本填满至少一个机架;否则规划出来会有"一个小众应用占着一整个机架却跑不满"的浪费。这其实和我们之前遇到的">=1 GPU per phase"整数地板问题是同一回事,只是粒度从一张卡上升到了一个机架——应用越小众、机架越大,这种碎片浪费越明显。另一条是**对预测的依赖**:整份蓝图都建立在那份需求预测之上,预测越不准,事先该留的余量(或者说该做的鲁棒性考虑)就越多——这同样是规划时就要权衡的事,而不是留给运行时去救。

## 收尾

把上面串起来:router 假设在规划阶段的本质,是把"要设计一种能同时消化二维长度分布混合的通用机架"这个难题,换成"为每一种应用各设计一种专用机架、再把机房按需求切成几块"的一组简单子问题。之前那三个让单一比例模型失效的因素——上下文长度、延迟 SLO、需求差异——一个都没有真正消失,但在规划时它们被分别收进了同质的机架池里:上下文变成了每种机架的一个常数,SLO 变成了每种机架的一档,各应用的需求占比变成了机架数的配额。每一个,都比"全部搅在一起"要好规划得多。

所以真要往下做,规划阶段最自然的两步是:一是给 `solve.py` 的 decode 曲线加上上下文参数,让 `T_max(C)=β/(C·k_v)` 能随应用变,从而为每一种应用算出它专属的机架配方;二是在外面再写薄薄一层"按预测需求把机架配额分给各应用"的逻辑。前者我们手上已经有实测曲线、改动很小,是性价比最高的一步。
