# 一次性灌满 vs 涓流控温 —— 固定工作量比 JCT

固定总工作量 = **N 个 8192² fp16 GEMM**,两种调度,比作业完成时间(JCT):

- **A · BURST(一次性灌满,不管散热)**:全丢进去连续跑。撞 83°C 热降频,SM 时钟塌到 ~250–560MHz,每个 GEMM 变慢,**但 100% 占空、零空闲**。
- **B · PACED 涓流(把任务切小块,一小口一口喂,控温)**:每喂一小块(4 个 GEMM)就按温度用积分控制器微调一个 ≤1s 的小停顿,把温度**稳稳钉在 81°C**(略低于 83°C 降频点)。**全程满频 1305MHz、从不降频**,代价是小停顿累积的空闲。

两者同基线预冷起跑、不锁频(看系统自己的热管理)。

```
burst_vs_paced/
├── run_experiment.py   # 跑 PACED涓流(冷启) + BURST,记录 timeline.csv + meta.json
├── plot_burst_vs_paced.py
└── timeline.csv  meta.json  fig_burst_vs_paced.png
```
运行:`CUDA_VISIBLE_DEVICES=0 python3 run_experiment.py && python3 plot_burst_vs_paced.py`(无需 sudo)

## 结果(Tesla V100-DGXS-32GB)

| 作业量 / PACED 策略 | BURST JCT | PACED JCT | 谁赢 |
|---|---|---|---|
| **N=2500,涓流控温(本次)** | 96.4 s | **94.7 s** | **PACED 险胜 1.02×** |
| N=800,先灌满-撞顶-狂冷(早期粗糙版) | **12.9 s** | 19.6 s | BURST 胜 1.51× |

**本次(涓流 + 长作业)PACED 反超**——但只快 1.8%,且 BURST 这次因预冷超时**带着 71°C 余温起跑**(几秒就降频、没吃到满频开局),属吃亏方。所以严格说这是**平局 / 交叉点**。

### 稳态数据(图 `fig_burst_vs_paced.png`)
- **PACED 涓流**:全程满频 1305MHz、温度钉在 81°C;稳态占空比 ~30%(63s 计算 + 62s 小停顿),有效 ~26 GEMM/s。
- **BURST**:时钟一路热浸透塌缩,从 boost 跌到 **最低 217–247MHz**(降 ~80%),有效从开局 ~40 跌到稳态 ~19 GEMM/s。

## 核心结论:存在交叉点,随作业变长天平从 BURST 倒向 PACED

| 作业长度 | 谁快 | 为什么 |
|---|---|---|
| **极短**(没烧到 83°C) | 平 | 都不降频 |
| **短**(~10s) | **BURST** | 只在尾段降频;PACED 的冷却空闲 > 降频损失 |
| **长**(~90s+) | **PACED 涓流** | BURST 深度热浸透塌到 ~250MHz,持续速率被压到 PACED 满频-占空速率之下 |

机制:BURST 的有效速率 ∝ 降频后的时钟,而**时钟随热浸透持续下滑**(越跑越慢,本次探到 217MHz);PACED 涓流锁住满频、速率只受占空比限制、**不随时间衰减**。作业越长,BURST 衰减得越多 → 越往后越吃亏。

> 关键:用户的**涓流**思路(切小块、每次只加一点、控温而非"狂冷")是 PACED 能追平的前提——它把空闲降到"刚好够散热"、并锁住满频。早期"灌满→撞顶→狂冷"的粗糙 PACED(空闲占 48%、还得大段冷却)则完败。

## 边界(诚实标注)
- 本次 PACED 仅快 1.8%,且 BURST 起跑带余温吃亏——**严格是平局**;要干净地证明交叉,需 BURST 也冷启动、并把作业再加长(N=5000+),预计 PACED 优势会拉开。
- 结果依赖本卡的**慢散热 + 深度降频**特性;散热好的卡 BURST 降频浅,BURST 会一直赢。
- PACED 目标温度越贴近 83°C,其 P_cool 越大、占空比越高、越快(本次用 81°C 留了 2°C 余量)。
